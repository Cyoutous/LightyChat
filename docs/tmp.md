《基于Socket的局域网聊天软件 需求规格说明书 V1.0》
1. 引言
1.1 项目背景
本项目旨在设计并实现一个轻量级、纯命令行的局域网聊天软件。软件不依赖任何第三方前端库或复杂框架，完全基于Socket API构建，以体现对TCP/IP协议栈、应用层协议设计及并发编程的深刻理解。

1.2 设计目标
极致轻量：开箱即用，CPU和内存占用极低。

全双工即时通信：用户可随时接收和发送消息，操作不阻塞。

二元状态管理：用户仅有“在线”和“离线”两种状态，逻辑清晰。

房主即服务器：聊天室的创建者既是服务器宿主，也是普通参与者，拥有管理特权。

2. 用户角色与核心概念
用户：所有连接到聊天室的客户端。标识符（ID）可以是自定义昵称或系统分配ID。

房主：创建聊天室的特殊用户。他是服务器的提供者，拥有踢人、禁言等管理权限，并通过同样的客户端界面输入管理指令。

聊天室：一个由房主创建、其他人通过IP:端口加入的临时会话空间。聊天室关闭后，一切在线数据不复存在。

3. 功能性需求
3.1 本地大厅与连接流程
启动后界面：软件以命令行界面启动，显示“本地大厅”菜单，提供选项：

创建聊天室

加入聊天室

修改默认配置（端口、聊天室名称等）

创建聊天室：

提示用户输入聊天室名称（输入“.”则使用默认名称）。
提示用户输入开放端口（输入“.”则使用默认端口）。
在后台启动服务器进程，并自动以房主身份连接到本机服务器。
界面显示“聊天室已创建，端口：XXXX”，并进入聊天界面，广播欢迎信息。
加入聊天室：

提示用户输入目标地址（IP或内网穿透后的中转地址）和端口。
提示用户输入自定义昵称（仅允许英文字母、数字、下划线）。
连接成功后，服务器端进行昵称处理（见3.2），返回实际ID。
客户端显示欢迎信息和实际ID，服务器广播“用户 [ID] 加入聊天室”。
3.2 用户标识符（ID）规则
自定义昵称：用户可输入仅含[A-Za-z0-9_]的字符串。

防重名机制：服务器检测到重名时，自动在原昵称后追加一串数字（如“_1”），并将最终的ID告知用户。

强制系统分配：房主可在创建聊天室时选择“自动分配ID”模式。此模式下，所有用户的自定义昵称被忽略，统一分配为User#XXX格式（XXX为数字编号）。

ID用于私聊：私聊指令中的目标对象即此系统最终确认的ID。

3.3 核心通信功能
公屏聊天：

用户在输入框输入不含/前缀的纯文本，按回车发送。

服务器收到后，向所有在线用户（包括发送者自己）广播该消息。

发送者的消息只有在服务器成功广播后，才出现在自己的屏幕上，以此确认发送成功。

私聊：

格式：/wsp <目标用户ID> <消息内容>

服务器仅将消息转发给发送者和目标用户。

若目标用户不在线，服务器立即向发送者返回明确错误：[系统] 消息发送失败：用户 ‘<ID>’ 不在线。

系统广播：

用户加入或退出（主动离开或因连接断开）时，服务器广播通知，如：[系统] 用户 ‘Alice’ 加入了聊天室。

文本支持：所有消息均为纯文本，支持特殊字符（不含控制字符）。

3.4 指令系统（以 / 开头）
通用指令：

/list：列出所有在线用户ID。

/whoami：查看自己当前的实际ID。

/help：显示可用指令列表及用法。

/quit 或 /exit：正常退出聊天室，返回本地大厅。

房主专用管理指令：

/kick <用户ID>：强制断开指定用户的连接。被踢者看到“您已被移出聊天室”。

/mute <用户ID>：禁言指定用户。被禁言者仍可接收消息，但发送公聊或私聊会失败，并收到“您已被禁言”的提示。

/unmute <用户ID>：解除禁言。

指令错误处理：若输入以/开头但非有效指令，系统提示“未知指令，输入/help查看帮助”。若指令参数错误，提示正确用法。

3.5 数据持久化（仅限房主）
当聊天室创建后，服务器开始将所有公聊消息和系统事件（加入、离开、踢人等）写入本地日志文件。

日志文件以时间戳或聊天室名称命名，每次创建聊天室生成新文件，不继承历史。

私聊消息不会出现在服务器日志中，以保证隐私。

普通用户不保存任何历史记录。

4. 非功能性需求
4.1 性能与规模
默认支持30人同时在线，流畅通信。

并发模型由开发者选择，但需在文档中论证其合理性。由于规模不大，简单多线程模型即可满足需求。

房主可在配置中调整最大连接数上限，软件不作硬性限制。

4.2 安全性
采用明文传输，不实现加密层。

基本协议防闯入：服务器仅处理符合本软件自定义协议格式的数据包。对于无法识别的非法连接请求或数据，直接断开连接并记录。能防止不知协议的外部程序误连。

4.3 可靠性
全双工与反馈：客户端发送线程与接收线程分离，确保消息接收不阻塞用户输入。所有消息的发送成功/失败状态，用户均能得到明确反馈。

异常断线处理：服务器能检测到客户端异常断开，并自动广播其离线消息，清理相关资源。

4.4 可维护性
代码架构清晰分层：网络通信层、协议解析层、业务逻辑层分离。

核心通信部分必须有详尽的注释，特别是TCP粘包处理和应用层协议定义的代码段。

5. 用户界面原型
纯命令行。

进入聊天室后，输入提示符统一为 [你的ID] >。

接收到的消息会实时打印在终端上。为避免干扰输入，消息打印不会截断已输入到一半的文本，而是换行显示后，在新行重新打印提示符和未发送的输入内容（此为推荐交互实现，需求阶段仅作方向性要求）。

6. 约束与假定
使用C/C++或Java等支持原生Socket API的语言。

通信基于TCP协议。

所有功能在局域网或可控的广域网环境下测试通过。

用户的操作系统和终端环境支持标准的命令行输入输出。
