# 协议规范 V1.0

## 1. 概述
本协议为“局域网聊天软件”的专有应用层协议，运行于 TCP 之上。它定义了客户端与服务器之间的消息格式、消息类型、长度限制及交互约定，确保通信双方能够正确地编码、传输、解析和路由聊天数据。

### 设计原则
- 二进制格式，紧凑高效，易于解析。
- 变长发送方与接收方ID，支持灵活的身份标识。
- 通过“总长度”字段解决TCP粘包/半包问题。
- 本地预处理与服务器兜底相结合的健壮性设计。

## 2. 消息格式
每条消息由固定头部、变长ID区、变长消息体组成。

```text
+-------------------------------------------------+
|       起始魔数 (2)      |       消息类型 (2)     | 
|                     总长度 (4)                  | 
|    发送方ID长度 (2)     |     接收方ID长度 (2)    | 
|     发送方ID (变长)     |   接收方ID (变长)       |    
|                  消息体 (变长)                   |
+-------------------------------------------------+
```

### 字段详细说明
| 字段 | 长度 | 类型 | 说明 |
| --- | --- | --- | --- |
| 起始魔数 | 4 | 无符号整数，大端 | 固定为 `0x4C 0x43 0x48 0x54` (ASCII: “LCHT”)。用于快速识别本协议报文，拒绝非法连接。 |
| 消息类型 | 2 | 无符号整数，大端 | 标识消息的业务类型，决定处理逻辑。取值见第3节。 |
| 总长度 | 4 | 无符号整数，大端 | 整条消息的总字节数，包含从魔数到消息体的所有字段。接收方解析出此值后，可准确切分出一条完整的消息，是解决粘包/半包的核心字段。合法范围：12 ~ 4096（见第4节）。 |
| 发送方ID长度 | 2 | 无符号整数，大端 | 后随发送方ID的字节数。为0表示无发送方ID（如心跳包）。 |
| 接收方ID长度 | 2 | 无符号整数，大端 | 后随接收方ID的字节数。特殊值 0 表示广播（所有人）；非0则后跟对应长度的接收方ID，用于单播。 |
| 发送方ID | 变长 | UTF-8字符串 | 发送者的实际ID。其长度由“发送方ID长度”字段指定。 |
| 接收方ID | 变长 | UTF-8字符串 | 目标用户的ID。仅当“接收方ID长度”非0时存在。若为0，则此区域不存在。 |
| 消息体 | 变长 | 字节序列 | 消息的有效载荷，UTF-8编码的文本内容。对于某些控制类消息（如心跳），长度可为0。 |

固定头部长度：14 字节。

消息体实际长度 = 总长度 - 14 - 发送方ID长度 - 接收方ID长度。

## 3. 消息类型

| 类型值 | 常量名 | 方向 | 描述 |
| ---- | ---- | ---- | ---- |
| 0x0001 | TYPE_MESSAGE | 客户端 → 服务器 | 客户端发出的聊天消息。通过接收方ID长度区分公屏（0）与私聊（非0）。 |
| 0x0002 | TYPE_COMMAND | 客户端 → 服务器 | 客户端发出的指令请求，指令名称和参数存储在消息体中。 |
| 0x0003 | TYPE_SYSTEM | 服务器 → 客户端 | 服务器发出的系统广播（加入、离开、踢出等）。 |
| 0x0004 | TYPE_MESSAGE_DELIVER | 服务器 → 客户端 | 服务器转发的聊天消息，可能是公屏广播，也可能是私聊单播。 |
| 0x0005 | TYPE_RESPONSE | 服务器 → 客户端 | 服务器对指令请求的响应。消息体首字节为状态码（0x00成功，0x01失败），剩余部分为响应内容（UTF-8）。 |
| 0x0008 | TYPE_HEARTBEAT | 客户端 → 服务器 | 心跳PING，通常发送方ID长度与接收方ID长度均为0，消息体为空。 |
| 0x0009 | TYPE_HEARTBEAT_ACK | 服务器 → 客户端 | 心跳PONG，格式同上。 |

## 4. 长度限制
最大消息长度：4096 字节。

### 限制策略
- 客户端：在打包消息前，计算完整消息的总长度。若超过4096字节，拒绝发送并提示用户“消息过长，请分段发送”。
- 服务器：接收并解析出“总长度”字段后，立即检查。若该值大于4096，直接断开连接，防止内存攻击。

说明：4096字节可容纳约1300个中文字符，满足正常聊天需求，同时避免大消息占用过多内存。

## 5. 典型交互示例

### 示例1：用户 Alice 发送公屏消息 Hello, 世界
**客户端打包：**
- 魔数：4C 43
- 消息类型：00 01 (TYPE_MESSAGE)
- 发送方ID长度：5（Alice UTF-8 长度为5）
- 接收方ID长度：0（广播）
- 发送方ID：41 6C 69 63 65
- 消息体：48 65 6C 6C 6F 2C 20 E4 B8 96 E7 95 8C（13字节）
- 总长度：12(头) + 5 + 0 + 13 = 30 = 0x1E

**服务器处理**：识别接收方ID长度为0，遍历在线用户表，向每个用户发送一条 `TYPE_MESSAGE_DELIVER` 消息（含发送方ID和消息体）。

**接收方显示**：[公屏] Alice: Hello, 世界

---

### 示例2：Alice 私聊 Bob 发送 Hi
**客户端打包：**
- 接收方ID长度：3（Bob UTF-8 长度为3）
- 接收方ID：42 6F 62
- 消息体：48 69（2字节）
- 总长度 = 12 + 5(Alice) + 3(Bob) + 2 = 22 = 0x16

**服务器处理**：接收方ID长度非0，查找 Bob 是否在线。若在线，将消息以 `TYPE_MESSAGE_DELIVER` 转发给 Bob（同时抄送 Alice）；若不在线，向 Alice 返回 `TYPE_RESPONSE`，状态码 0x01，内容：用户 'Bob' 不在线。


## TMP
#### 6.1 为什么采用“总长度”字段解决粘包？
TCP是流协议，发送端多次 send 可能被合并为一个TCP段，或一个 send 被拆分成多个 recv 接收。若依赖消息边界符，需保证消息体不包含该字符；而本协议通过“总长度”让接收方明确知道消息边界，是一种通用、可靠且高效的方案。

#### 6.2 魔数的多重价值
快速识别非法连接：外部程序（如浏览器、扫描工具）误连时，服务器可立即断开，避免无效解析。

辅助调试：在抓包时，可快速过滤出本协议报文。

成本极低：仅2字节。

#### 6.3 接收方ID长度为0的特殊约定
将“广播”定义为接收方ID长度为0，而非用特殊字符串（如*），使得：

协议头结构不变，处理逻辑统一（判断长度即可）。

无歧义，避免了用户昵称可能与广播标识冲突的风险。

#### 6.4 本地指令预处理与协议无关
/quit、/clear 等指令在客户端本地字典中匹配并执行，不产生网络流量。这不仅减轻了服务器负担，也提升了响应速度，并避免了网络异常时无法退出等问题。协议仅需承载需服务器处理的指令。

#### 6.5 消息长度限制的层次化防御
第一道防线（客户端）：本地拦截，直接告知用户，节省带宽。

第二道防线（服务器）：即使客户端被恶意修改，服务器也会根据协议规范中的上限值拒绝超大消息，保护服务端内存安全。这种防御性设计是网络软件健壮性的重要体现。

#### 6.6 心跳机制与僵尸连接清理
客户端定期发送空 PING，服务器回复 PONG。若服务器超过30秒未收到任何数据，则主动断开该连接并广播离线通知。这种设计能够及时回收被异常断开的客户端所占用的资源，是长连接系统不可或缺的组成部分。
