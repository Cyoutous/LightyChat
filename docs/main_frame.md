# 概要设计框架
## 1. 系统架构图
整体采用经典的 客户端-服务器 (C/S) 架构。
```
graph TD
    subgraph 客户端进程
        CLI[命令行交互模块]
        CLISend[发送线程]
        CLIRecv[接收线程]
        CLI <--> CLISend
        CLI <--> CLIRecv
    end

    subgraph 服务器进程
        Acceptor[连接接收模块]
        ClientPool[客户端管理模块]
        MsgRouter[消息路由模块]
        CmdHandler[指令处理模块]
        Logger[日志模块]
        Acceptor --> ClientPool
        ClientPool <--> MsgRouter
        MsgRouter <--> CmdHandler
        MsgRouter --> Logger
    end

    CLISend -- TCP Socket --> Acceptor
    CLIRecv -- TCP Socket --> MsgRouter
```

### 核心思路：
- 客户端是“收发分离”的双线程模型。
- 服务器拆成五个模块，各司其职。所有消息流转都经过消息路由模块这个中枢。

## 2. 模块职责简述
### 客户端
| 模块	| 一句话职责 |
|:-----|:---------|
|命令行交互模块	|接管用户的所有输入输出，把输入交给发送线程，把接收线程收到的消息打印出来。|
|发送线程	|等待用户输入，打包成协议格式，通过Socket发出去。|
|接收线程	|死循环从Socket收数据，解包，交给交互模块显示。|

### 服务器
| 模块	| 一句话职责 |
|:-|:--|
|连接接收模块	|监听端口，accept新连接，把新socket注册到客户端管理模块。|
|客户端管理模块	|维护在线用户表（ID、socket、状态），提供增删改查接口。|
|消息路由模块|	所有消息的转发中枢。从某个socket收到一条完整消息后，判断类型，决定投递范围。|
|指令处理模块|	专门解析和处理以/开头的指令。由消息路由模块调用，结果返回给路由模块进行单播。|
|日志模块|	接收消息路由模块发来的“可记录事件”，写入文件。仅服务器端存在。|

## 3. 核心数据流（最简路径）
以一条公屏消息为例：

```
text
用户A输入 "hello"
    -> CLI将"hello"交给发送线程
    -> 发送线程按协议打包，send()
    -> 服务器接收线程recv()到数据，解包出完整消息
    -> 消息路由模块判断：类型为"公屏消息"
    -> 路由模块遍历在线用户表，把这条消息转发给每一个用户（包括A自己）
    -> 日志模块记录这条公屏消息
    -> 所有在线客户端的接收线程收到消息，解包，打印到屏幕
```
## 4. 服务器并发模型 【待定】
这里需要你做一个选择。我给出两种方案：

|方案|	思路|	优点|	缺点|
|:-|:-|:-|:-|
A. 多线程模型|	每接受一个连接，开一个新线程专门负责这个socket的收发。|	编程思路直接，每个线程逻辑简单。|	线程数=用户数。对30人来说毫无压力。|
|B. IO多路复用|	单线程（或少量线程）用select/epoll同时监听所有socket。	|高性能，C10K级方案。报告里更“高级”。	|代码结构更复杂，状态管理需要自己维护。


我推荐方案A。

如果你有不同意见，我们再讨论。

## 5. 协议消息类型 【待定】
所有消息在应用层至少需要区分为以下几类。具体格式暂不定义，先确认分类是否完整：

- 系统消息（如加入/退出通知）
- 公屏聊天
- 私聊
- 指令请求（客户端发来的/xxx）
- 指令响应（服务器返回给指令发出者的结果）
- 心跳消息
